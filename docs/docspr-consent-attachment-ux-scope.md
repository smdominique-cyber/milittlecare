Context — the real-world workflow this serves (ground the design here): a home daycare provider handles compliance in small moments, not batch sessions. Two physical contexts: (a) enrollment at the kitchen table on their phone — a family signs paper forms, the provider records "on file" and photographs the signed paper then and there; (b) admin later at a computer — batch entry, file uploads. The UX must serve both. Guiding principle from childcare recordkeeping best practice: make each small moment frictionless so paperwork doesn't pile up. The current flow breaks this — recording a consent kicks the provider out of the modal, forcing them to navigate back in to attach the signed form, which is fatal for the "record it and snap the paper right now" motion.
Scope — three things, all UI/UX (no schema change expected):
1. Fix the save-exits-modal bug across the consent + intake modals (the core fix). Replicate the proven MedicationModal save-confirmation pattern in EnrollmentConsentsModal and ChildIntakeModal: on a successful save (recording a consent, capturing intake acks), the modal STAYS OPEN, shows an inline ? confirmation, and the list/state updates in place — the provider is NOT kicked back to the family tab. Diagnose the close mechanism first and report it (likely the parent onSaved?refresh unmounting the modal, same as medication was — confirm it's the same across both modals or flag if they differ). The workflow payoff: because the modal stays open, the just-recorded consent's attachment slot now appears in place, so the provider can immediately attach the signed form — "record ? ? ? attach" in one motion, no navigation.
2. Camera capture on mobile. In ConsentAttachmentSlot, the file input should let a phone user open the rear camera directly (capture="environment") to photograph a signed form, in addition to the existing file picker. On desktop, capture is ignored — falls back to file picker, same component. Address: (a) HEIC handling (the validator already allows HEIC, but iPhone photos run 5-8MB against the 10MB cap); (b) whether to add client-side image compression before upload (the messages feature already uses browser-image-compression — propose reusing it; phone photos of documents are large and a high-res or multi-page shot could exceed the cap); (c) a "retake" affordance for a bad/skewed snap. Propose the cleanest approach; flag if compression meaningfully expands scope or needs a new dependency decision.
3. Cross-modal consistency. Ensure all three record modals (intake, consents, meds) behave IDENTICALLY on save: stay open, inline ?, list updates in place, sub-forms reset, modal closes only on explicit close. Medication already does this — make intake and consents match it exactly, one consistent pattern, not three variations. Note any place a modal legitimately needs to differ and why.
Also propose (the open design question, delegated to you): the envelope-vs-per-form attach choice. The data model supports both (polymorphic target — an attachment can point at the intake-bundle envelope ack OR a specific consent ack). A provider should be able to attach a scan of one specific signed form AND/OR a scan covering a whole signed packet. Propose the cleanest per-surface default — e.g. per-consent attach in the enrollment/medication modals, envelope-level "whole signed packet" attach in the intake bundle, or some mix — and justify it against the two physical workflows (phone-at-table vs. desktop-admin). Lay out the tradeoff (more flexibility = more UI clutter); recommend, don't over-build. Seth will react to the proposal.
